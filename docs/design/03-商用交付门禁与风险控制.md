# 03 商用交付门禁与风险控制

## 1. 门禁目标

1. 避免“脚本成功=可以商用”的误判。
2. 把风险评估前置到交付阶段。
3. 形成可追溯的证据闭环。

## 2. 门禁清单

| 维度 | 必要项 |
|---|---|
| 代码兼容 | QuecPython 兼容检查通过 |
| 设备验证 | smoke 通过且关键路径可复现 |
| 稳定性 | soak 达到最低时长要求 |
| 安全性 | 无敏感信息泄露、风险命令受控 |
| 文档 | 部署步骤、回滚方案、风险说明齐全 |

## 3. 风险分级

| 等级 | 定义 | 处理 |
|---|---|---|
| P0 | 阻断上线风险 | 必须停止交付 |
| P1 | 高概率业务影响 | 修复后复验 |
| P2 | 可接受但需跟踪 | 带风险说明发布 |
| P3 | 低影响优化项 | 进入后续迭代 |

## 4. 证据要求

1. 原始日志文件
2. JSON 结构化报告
3. 版本对比信息（固件/脚本）
4. 人工复核结论
